# Agentic RAG 架构说明

## 为什么不是普通 RAG

普通 RAG 往往是“检索几段资料 -> 塞进 prompt -> 生成回答”。模拟面试官的问题不只是回答知识，而是要判断候选人简历里哪些点最可能被问挂。因此本项目采用更窄的 Agentic RAG：

1. 检索不是为了回答用户，而是为了规划下一轮追问。
2. 资料卡里保留的是追问模板、反模式和评分证据，不是长文内容。
3. 每轮生成后做 critic，自检是否绑定简历证据、资料依据和技术细节。
4. 前端展示依据，让用户知道这一轮为什么问这个问题。

## 运行时链路

```text
简历 + 目标岗位 + 场景 + 最近问答
        |
        v
轻量检索器：keywords + tags + scenario boost
        |
        v
资料卡 Top-K + 简历证据 + 风险假设
        |
        v
deepseek-v4-pro 生成追问
        |
        v
critic 检查：泛泛/无依据/多问题/无技术关键词
        |
        +-- 不合格：带问题重写一次
        |
        v
返回问题 + source cards + tags + evidence
```

## 当前取舍

- 不引入向量库：资料卡数量小，关键词/tag 检索足够解释、稳定、易测。
- 不引入 LangGraph：当前只有一个主 workflow，显式 Python 编排更快。
- 不保存用户简历：文件只在内存中解析，前端状态本地持有。
- 不提交原始资料：只提交整理后的 source cards 和来源索引。

## 风险

- 资料卡数量少时，覆盖面依赖人工整理。
- OCR 中文识别质量取决于 Tesseract 语言包和 PDF 清晰度。
- Critic 目前是启发式规则，不是严格结构化评估器。

## 后续

- 增加岗位 JD 资料卡和岗位画像。
- 将 critic 输出结构化，前端展示“为什么这题有训练价值”。
- 加入用户回答质量评分器，把 source cards 映射到复盘证据。
