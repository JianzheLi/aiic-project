# Product Memo 草稿

## 1. 目标用户

首版目标用户是准备互联网大厂或 AI 相关技术实习面试的中国本科生，尤其是有项目但不确定能否经得起追问、正在背八股但不会和项目结合、做过 RAG/Agent/LLM demo 但担心被问“是不是只调 API”的学生。

## 2. 用户痛点

公开面经、岗位 JD、竞品和项目样本调研显示，学生不缺资料和题库，真正缺的是低成本、高频、接近真实面试的连续追问训练。真实面试往往从项目经历切入，再追问技术选型、个人贡献、指标验证、失败路径和迁移能力。通用聊天机器人可以解释概念，但不会稳定地抓住用户回答里的漏洞并持续追问；题库平台资料丰富，却无法针对用户自己的项目做动态复盘。

## 3. 产品方案

AI 模拟面试官把首版闭环收敛为三步：

1. 用户选择训练场景并粘贴项目经历。
2. AI 面试官围绕项目连续追问 3-5 轮。
3. 用户结束后获得结构化复盘，明确最可能被问挂的点和下一轮行动。

首版固定三个场景：项目深挖压力面、后端八股项目化追问、RAG/Agent 项目真实性拷打。这三个场景覆盖了调研中最常见、最容易暴露学生短板的面试路径。

## 4. 为什么不是普通聊天机器人

普通聊天机器人默认顺着用户提问回答，很容易变成知识讲解或泛泛建议。本产品用轻量状态机控制面试阶段、轮次和结束条件，用场景化 prompt pack 约束模型角色：先做面试官提问，再做追问者深挖，最后做教练复盘。这样用户获得的是一轮训练闭环，而不是一次聊天。

## 5. MVP 取舍

我们没有做登录、数据库、语音、视频、完整简历解析或复杂 agent 框架。原因是 16 小时挑战的关键不是堆功能，而是做出最小但有训练价值的闭环：贴项目、被追问、暴露挂点、知道下一步怎么补。

## 6. 技术实现

当前实现沿用 FastAPI + Vite/React + Docker Compose。后端新增 `/interview/message`，使用 OpenAI-compatible Chat Completions 接入 DeepSeek 等模型；前端改造为面试训练工作台，通过 `/api/*` 同源代理访问后端。测试覆盖后端状态推进和前端核心交互，公网 demo 以 `http://8.139.254.60:3000/` 为验收入口。

## 7. Demo 亮点

用户粘贴一段 RAG 项目经历后，AI 不会只问“请介绍项目”，而是直接抓住 chunk、embedding、坏例分析、评估和部署追问。结束复盘会指出项目可信度、专业深度、表达结构、工程闭环、承压表现等维度上的具体挂点。

## 8. 后续方向

后续可以加入 JD 匹配、简历解析、历史训练记录、语音模拟和更严格的结构化评分。等需要 tools/session/tracing 时再考虑 OpenAI Agents SDK；等需要长期状态和 human-in-the-loop 时再考虑 LangGraph。
