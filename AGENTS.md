# AGENTS.md

## 项目角色

你在一个远程 Ubuntu 服务器上，负责部署、调试和迭代 AI 产品。当前项目从“最小可用全栈聊天样本”开始，目标是迭代成面向中国本科生的“AI 模拟面试官”产品，用于“16 小时 AI Agent 产品挑战”。

## 协作约定

- 默认使用中文沟通，代码标识、环境变量和命令保持英文。
- 优先保证快速部署、稳定运行、方便迭代。
- 不提交真实密钥；`.env` 必须保留在本地。
- 前端默认端口是 `3000`，后端默认端口是 `8000`。
- 暂时不引入 nginx、数据库、登录系统或复杂服务编排。
- 修改前先理解当前目录职责，保持代码结构简单，避免过度抽象。
- 优先查看可用 skill，提升实现和排查效率。
- 默认先对齐 `target/` 下的挑战材料与计划文档，再开始大规模编码。
- 讨论中的产品决策、范围取舍和执行步骤统一沉淀到 `target/PLAN.md`。
- 原始挑战说明 PDF 只作为输入材料保留，不直接改动。

## 当前挑战方向

- 产品主题：AI 模拟面试官。
- 目标市场：中国学生求职与升学面试准备场景。
- 初始目标用户：准备大厂实习面试或保研复试的本科生。
- 核心目标：做出一个比直接使用通用聊天机器人更有训练价值的最小产品闭环。
- 当前产品结构：八股知识点、简历经历、手撕代码、完整模拟四个训练入口，分别维护问题来源、上下文和反馈维度。
- 产品优先级：真实痛点 > 核心闭环 > 可访问部署 > 界面精美 > 功能数量。
- MVP 阶段优先做文字交互；语音、视频、登录、数据库和复杂编排都不是默认必需项。

## 计划维护

- `target/2026-05-09_项目挑战说明.pdf`：原始挑战说明。
- `target/PLAN.md`：当前产品计划、讨论问题、执行清单和提交清单。
- 每完成一个重要取舍或功能迭代，都要同步更新 `target/PLAN.md`。
- 计划文档需要区分“已决定”“待讨论”“后续可选”，避免把想法误写成承诺。

## 部署约定

- Docker Compose 是首选部署方式。
- 云服务器 demo 访问方式为 `http://服务器公网IP:3000`。
- 当前公网 demo URL 为 `http://8.139.254.60:3000/`。
- 浏览器侧只依赖 `3000` 端口，前端服务通过 `/api/*` 同源代理访问后端。
- 后端健康检查为 `http://服务器公网IP:8000/health`。
- 后续 OpenAI-compatible API 通过环境变量接入。

## Git 控制
- 在合适的一个功能完成后，需要提交 git commit，并且 push 到远程上。
- 挑战评分会关注 commit history，避免把所有变更压到最后一次提交。
